Git学习笔记（一）

发表于 2017-04-03 | 分类于 版本控制工具

前言:在去年这个时候就已经开始使用git来管理代码了。但仅仅只会平时开发中经常遇几个git命令，对于其中的具体工作原理不大清楚。所以当遇到问题时，解决思路经常是模糊的。所以就抽时间认真地学习了一遍。俗话说的好，最好的学就是教。本文是参考了前辈们的文章根据自己理解写的一个git学习总结,有出错的地方希望大家多多纠正。

一、关于版本控制

在介绍Git之前我们需要了解一下什么是版本控制。

版本控制系统（VCS）最基本的功能就是版本控制。所谓的版本控制，意思就是在文件的修改历程中保留修改历史，让你可以方便的查看或者撤销之前对文件的修改操作。

因为我们在开发过程有时会出错，比方说现在你写的代码有点问题，想退回改动之前的样子， 但是改动的地方太多，肯定是记不住的。那么这个时候就需要有一个版本控制系统帮助你恢复到之前的代码版本。
用过Photoshop的都知道，我们可以通过ctrl+z（或者ctrl+alt+z）回退到某一个历史记录中。版本控制系统功能和PS的回退功能类似，只不过PS是软件会自动记录我们的操作，而版本控制系统需要我们主动提交修改的版本。
版本控制系统根据不同的工作需求分为下面几类。

1、本地版本控制系统

许多人习惯用复制整个项目目录的方式来保存不同的版本，或许还会改名加上备份时间以示区别。这么做唯一的好处就是简单。不过也会带来很多问题比如：有时候会混淆所在的工作目录，一旦弄错了文件丢了数据撤销恢复就麻烦了。
为了解决这个问题，很早以前就开发了许多 本地版本控制系统，其中最流行的一种叫做RCS。这种系统都是采用某种简单的数据库来记录文件的历次更新差异。

本地版本控制系统适合于个人开发，对于多人协作开发就不适合了。

2、集中式版本控制系统

大多数情况下开发一个项目我们是需要多人协同开发的，那么本地版本控制系统就不适合了。这时候我们可能会选择集中式版本控制系统（Centralized VCS）。
工作模式：
假设你在一个人团队，你们计划开发一个软件，并决定使用集中式版本控制系统来管理你们的代码。于是：

  1. 作为项目的主工程师，首先你一个人花了两天的时间搭建了项目框架；
  2. 然后，你在公司的服务器（这个服务器可以是公司内的设备，也可以是你们买的云服务）上 创建了一个中央仓库，并把你的代码提交到了中央仓库上。
  3. 你的两个队友 从中央仓库中取到你的初始代码，从此刻开始，你们三个人开始 并行开发。
  4. 在之后的开发过程中，你们三个人为了工作方便，总是每个人独立负责开发一个功能，在这个功能开发完成后，这个人就把他的这些 新代码提交到中央仓库。
  5. 每次当有人把代码提交到中央仓库的时候，另外两个人就可以选择 把这些代码同步到自己的机器上，保持自己的本地代码总是最新的。

这样，你们三个人在各种电脑上同步开发一个项目，互相不受影响。只需通过中央服务器进行交流。
当然，实际开发工作中不会这么简单，因为你时常会需要处理代码冲突，查看版本历史，回退代码版本等等。
另外，由于集中式版本控制系统的最新代码是存储在中央服务器的，当如果中央服务器出现了故障，那么最新版本的数据就很有可能丢失。
其次，这种工作模式需要在联网的情况下工作，因为你每次需要最新版本的代码时都需要从中央服务器中下载下来。

SVN就是集中式版本控制系统的典型代表。

3、分布式版本控制系统

分布式管理系统（Distributed VCS/DVCS）和集中式版本控制系统的区别在于，分布式版本控制系统没有真正的中央仓库，每个电脑都是一个节点，每个节点都保存着所有版本历史。每个人在自己的机器上就可以提交代码，查看历史，而无需联网和中央服务器进行交互。因为每台电脑上都会有一个本地仓库。如果有人修改了项目中的某一部分代码，它可以将修改的部分推送给其他队员。

大部分情况下为了管理和协调团队的交流，分布式版本控制系统会设立一个中转站（代码托管平台，比如github）。


工作模式
依然以三人团队为例

  1. 首先，你作为主工程师，独立搭建了项目架构，并把这些代码提交到了本地仓库
  2. 然后，你在服务器（或者代码托管平台）上创建了一个中央仓库，并把1中的提交 从本地仓库推送到了服务器的中央仓库
  3. 其他同事把 中央仓库的所有内容克隆到本地，拥有了各自的本地仓库，从此刻开始，你们三人开始并发开发。
  4. 在之后的开发过程中，你们三个人总是每人独立负责开发一个功能，在这个功能开发过程中，一个人会把它的每一步改动提交到本地仓库。注意：由于本地提交无需立即上传到中央仓库，所以每一步提交不必是一个完整功能，可以是功能中的一个步骤或一个模块。
  5. 当一个人把某个功能开发完成之后，他就可以把这个功能相关的所有提交 从本地仓库推送到中央仓库。
  6. 每次当有人把新的提交推送到中央仓库的时候，另外两个人就可以把 这些提交同步到自己的机器上，并把它们和自己的本地代码合并。

使用分布式版本控制系统，多数的操作可以在本地进行，所以在无需联网的情况下可以提交代码、查看历史。
但是每个机器都有完整的本地仓库，所以初次获得项目时会比较耗时。而且每个机器都有完整的本地仓库，所以本地占用的空间比较大。

Git就是分布式版本控制系统的典型代表

二、Git工作原理

1.Git工作区域

  • 本地仓库区（History）：本地仓库中存储着你提交的所有版本的数据，你初次从中央服务器中克隆下的代码就存储在这里，同时你提交的新代码也会存储在这里。
  • 暂存区（Index/Stage）:用于临时存放你将要提交的新数据。
  • 工作区（Workspace）：就是你平时存放项目代码的地方
  • 远程仓库（Remote Directory）：就是托管代码的服务器。你可以与其进行数据交换

2.Git整体工作流程

针对不同的操作，git工作流程可能会有一些差异，我们这节先说一下整体的主工作流程，具体的操作对应的工作流程后面再具体介绍。
请看下图（可以暂时先忽略图中的git操作命令）

整体的工作流程是这样的：

  1. 程序员在本地的工作目录中对项目文件进行增、删、改等操作
  2. 可以将需要进行版本管理的文件放入暂存区,也可以将其直接提交到本地仓库
  3. 将暂存区中的文件提交大本地仓库中。
  4. 最后将本地仓库中的代码推送到远程仓库
    处于不同阶段的代码会有不同的状态：
  • 已经修改（modified）
  • 已经暂存（staged）
  • 已经提交（commited）

三、Git的安装和配置

1.下载安装Git

在Git官网中找到适合自己电脑系统的Git进行下载。下载后安装就一直按照默认的安装步骤操作即可。如果需要特殊配置仔细阅读配置选项。关于Git的安装这里不多介绍了，相信都会完成的。

2、Git配置

使用git config -l就可以查看现在的git环境详细配置。（下面的git命令都是在windows系统下的命令）。

查看不同级别的配置文件：

  • 查看系统配置：git config –system –list
  • 查看当前用户配置：git config –global –list
  • 查看当前仓库配置信息：git config –local –list
    Git相关的三个配置文件：
  1. /ect/gitconfig:包含了适用于系统所有用户和所有项目的值,–system系统级的（C:\Program Files\Git\mingw64\ect\gitconfig）
  2. ~/.gitconfig:只适用于当前登录用户的配置。（Win:C:\Users\Administrator.gitconfig） –global全局
  3. 位于git项目目录中的.git/config：适用于特定的git项目目录。（Win:C:\gitProject）,–local当前项目。
    注意：对于同一项配置，三个配置文件的优先级是1<2<3

3、设置用户名与邮箱（用户标识，必要）

当你安装Git后首先要做的事情是设置你的用户名称和e-mail地址。这是非常重要的，因为每次Git提交都会使用该信息。它会被永远的嵌入到你的提交中：

git config –global user.name “xxxxxx” #用户名
git config –global user.email xxxxxxxx #邮箱

只需要做一次这个个设置，如果你传递了–global选项，因为Git将总是会使用该信息来处理你在系统中所做的一切操作。如果你希望在一个特定的项目中使用不同的名称和e-mail诋毁，你可以在该项目中运行命令而不要–global选项。总之–global为全局配置，不加则为某个项目的特定配置。

4、添加或删除配置项

添加配置项：git config [–local|–global|–system] section.key value
其中[–local|–global|–system]是可选的，对本地，全局，系统不同级别的设置
section.key是区域下的键
value是对应的键值
例如我们要在student区域下添加一个名称为height值为198的配置项：
git config –system student.height 198

删除配置项：git config [–local|–global|–system] –unset section.key
比如将系统级的height配置项移除：
git config –system –unset student.height
还有其他更多的配置项，可以通过git config –list来查看。
