git学习笔记（四）

发表于 2017-04-03 | 分类于 版本控制工具


git远程仓库也是我们平时开发工作中重要的一块。我们经常用到的代码托管平台就是github，gitLab也经常会用到。

关于远程仓库没有什么过多的解释的，它和本地仓库没有什么区别，只是一个放在远程服务器的仓库。因为我们不可能一直把代码存放在本地电脑中，万一哪一天你的硬盘坏了，后果不堪设想。额，其实在我不懂事的年纪就吃过这样的亏，当时真的想抱着电脑跳护城河了（无奈）。
所以学会怎么使用远程仓库你就不用担心这些问题了。
本文介绍的更多是如何通过命令来完成本地仓库和远程仓库之间的交流。嗯~，看来还是得放图。

可见远程仓库的相关操作无非就是推送、拉取、建立追踪、查看、移除，添加等常用的操作。

当然在这之前你要学会怎么使用github，用法很简单，都有相关教程,看官网就可以了。

一、git clone命令

在工作过程中，一个团队项目，通常是主工程师先在远程服务器中创建一个版本库，然后队员把这个远程版本库克隆到本地电脑上，然后进行开发。这时就要用到了git clone命了。
语法格式：

$ git clone <版本库的网址>

执行完之后会在本地电脑中生成一个与远程版本库同名的目录。如果要指定不同的目录名，可以通过git clone中的第二个参数改变。

$ git clone <版本库的网址> <本地目录名>

git clone支持多种协议：

$ git clone http[s]://example.com/path/to/repo.git/
$ git clone ssh://example.com/path/to/repo.git/
$ git clone git://example.com/path/to/repo.git/
$ git clone /opt/git/project.git
$ git clone file:///opt/git/project.git
$ git clone ftp[s]://example.com/path/to/repo.git/
$ git clone rsync://example.com/path/to/repo.git/

其中git协议下载最快。ssh协议还有另外一种写法：

$ git clone [user@]exapmle.com:path/to/repo.git

ssh协议用于需要用户认证的场合，各种协议的区别请查看官网

我在我的github上新建了一个名叫gitTest的远程仓库，现在使用git clone命令克隆到本地。

发现在本地的当前目录下多个一个gitTest目录，进入到这个gitTest目录。

二、git remote

git remote命令用于查看远程主机的名称，接着上一节，我们在gitTest目录下执行git remote命令。

1.git remote-v

可以使用 git remote -v 查看远程主机的网址。

有两个地址，fetch是读，push是写。

2.git remote -o

在克隆版本库的时候，所使用的远程主机自动被git命名为origin。如果想用其他的主机名，需要用git clone命令的-o选项指定。

上面命令表示，克隆的时候，指定远程主机叫做test。

3.git remote show <主机名>

git remote show命令加上主机名，可以查看该主机的详细信息。

4.git remote add

git remote add <主机名> <网址> 用于添加远程主机，有时候我们在本地建立好了本地仓库，并且把代码都提交到本地仓库中了，想把本地仓库与远程仓库建立
连接，将这些代码推送到远程仓库中。这个时候就用到git remote add 命令了。

三、git fetch

在团队开发中，如果别的队员向远程主机的仓库中提交最新的代码，或者主工程师对master分支进行了修改，这时候远程主机的仓库有了更新，我们需要将这些跟新取回本地，就要用到git fetch命令。

$ git fetch <远程主机名>

上面命令将远程主机的更新，全部取回本地。
git fetch命令通常用来查看其他人的进程，因为它取回的代码对你本地的开发代码没有影响。
默认情况下，git fetch取回所有分支（branch）的更新。如果只想取回特定分支的更新，可以指定分支名。

$ git fetch <远程主机名> <分支名>

比如，取回origin主机的master分支。

$ git fetch origin master

所取回的更新，在本地主机上要用“远程主机名/分支名”的形式读取。比如origin主机的master，就要用origin/master读取。
可以使用前面加到到git branch -a命令查看远程和本地分支。
取回远程主机的更新以后，可以在它的基础上，使用git checkout命令创建一个新的分支。

$ git checkout -b newBranch origin/master

此外，也可以使用git merge命令或者git rebase命令，在本地分支上合并远程分支。

$ git merge origin/master
$ git rebase origin/master

表示在当前分支上合并origin/master

四、git pull

git pull表示取回远程主机某个分支的更新，再与本地的指定的分支合并。它的完整格式如下：

$ git pull <远程主机名> <远程分支>：<本地分支名>

比如，取回origin主机的next分支，与本地分支master分支合并，需要这样写：

如果远程分支是与当前分支合并，则冒号后面的部分可以省略。

上面命令表示，取回origin/next分支，在与当前分支合并。实际上，这等于先做git fetch，再做git merge

在某些场合，Git会自动在本地分支与远程分支之间，建立一种追踪（tracking）。比如在git clone的时候，所哟地址分支默认与远程主机的同名分支建立追踪关系，也就是说，本地的master分支自动“追踪”origin/master分支。

git也允许手动建立追踪关系：

$ git branch –set-upstream mmaster origin/next

上面的命令指定master分支追踪origin/next分支。
如果当前分支与远程分支存在追踪关系，git pull就可以省略远程分支名。

$ git pull origin

上面命令表示，本地的当前分支自动与对应的origin主机“追踪分支”（remote-tracking branch）进行合并。
如果当前分支只有一个追踪分支，；连远程主机名都可以省略。

$ git pull

上面命令表示，当前分支自动与唯一一个追踪分支进行合并。

如果合并需要rebase模式，可以使用–rebase选项。

$ git pull –rebase <远程主机名> <远程分支名>:<本地分支名>

如果远程主机删除了某个分支，默认情况下，git pull不会在拉去远程分支的时候，删除对应的本地分支。这是为了防止，由于某人操作了远程主机，导致git pull不知不觉删除了本地分支。
但是，你可以改变这个行为，加上参数-p就会在本地删除远程已经删除的分支。

$ git pull -p

等同于下面的命令

$ git fetch –prune origin
$ git fetch -p

五、git push

git push命令用于将本地分支的更新，推送到远程主机。它的格式与git pull命令相仿。

$ git push <远程主机名> <本地分支名>:<远程分支名>

注意，分支推送顺序的写法是<来源地>：<目的地>，所以git pull是<远程分支>:<本地分支>，而git push是<本地分支>:<远程分支>。

如果省略远程分支名，则表示将本地分支推送与之存在“追踪关系”的远程分支（通常两者同名），如果该远程分支不存在，则会被新建。

$ git push origin master

上面的命令表示，将本地的master分支推送到origin主机的master分支。如果后者不存在，则会被新建。

如果省略本地分支名，则表示删除指定的远程分支，因为这等同于推送一个空的本地分支到远程分支。

$ git push origin :master

#等同于
$ git push origin –delete master

上面命令表示删除origin主机的master分支。

如果当前分支与远程分支之间存在追踪关系，则本地分支和远程分支都可以省略。

$ git push origin

上面命令表示，将当前分支推送到origin主机对应的分支。
如果当前分支只有一个追踪分支，那么主机名都可以省略。

$ git push

如果当前分支与镀铬主机存在追踪关系，则可以使用-u选项指定一个默认主机，这样后面就可以不加任何参数使用git push。

$ git push -u origin master

上面命令将本地的master分支推送到origin主机，同时指定origin为默认主机，后面可以不加任何参数使用git push了。

不带任何参数的git push，默认只推送当前分支，这叫做simple方式。此外，还有一种matching方式，会推送所有对应的 远程分支的本地分支。Git 2.0版本呢之前，默认采用matching方法，现在改为默认采用simple方式。如果修改这个设置，可以采用git config命令。

$ git config –global push default matching

或者

$ git config –global push default simple

还有一种情况，就是不管是否存在对应的远程分支，将本地的所有分支都推送到远程主机，这时需要使用–all选项。

$ git push –all origin

上面命令表示，将所有本地分支都推送到origin主机。
如果远程主机的版本比本地版本更新，推送是Git会报错，要求先在本地做git pull合并差异，然后再推送到远程机。这时，如果你一定要推送，可以使用–force选项。（表示强制推送）

$ git push –force origin

上面命令使用–force选项，结果导致远程主机上的更新的版本被覆盖。除非你很确定要这样做，否则应该尽量避免使用–force选项。
最后，git push不会推送标签（tag），除非使用–tags选项。

$ git push origin –tags
